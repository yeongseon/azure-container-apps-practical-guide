# Evidence Pack Provenance

This directory contains the raw evidence artifacts for the `acr-pull-failure` lab. The lab is a **deployment-level failure attribution + recovery validation** test of the Azure Container Apps behavior when a Container App is provisioned with an image tag that does not exist in its referenced Azure Container Registry. A single Container App is provisioned via `az deployment group create` with `image: '${ACR_LOGIN_SERVER}/labacr:does-not-exist'`, the Bicep deployment exits `Failed` with a per-operation error message containing `MANIFEST_UNKNOWN: manifest tagged by "does-not-exist" is not found`, the Container App resource is left in `provisioningState=Failed` with `latestRevisionName=null` and zero revisions, and the failed-deployment cohort is captured in 6 deployment-level evidence files by `trigger.sh`. The lab then issues `az acr build --image labacr:v1` followed by `az containerapp update --image <acr>/labacr:v1`, polls revision health until `Healthy + trafficWeight=100`, and captures the post-fix cohort plus 10 HTTPS probes against the recovered FQDN in 9 more evidence files by `fix-and-capture.sh`. The cohort was captured in a single live-Azure window on **2026-06-22** against RG `rg-aca-lab-acr` (Korea Central, azure-cli 2.79.0, containerapp extension 1.3.0b4 preview). All files are PII-scrubbed (subscription/tenant GUIDs replaced with the zero-GUID placeholder, employee aliases replaced with `demouser`, employee emails replaced with `user@example.com`, Log Analytics workspace customerId replaced with the zero-GUID placeholder); the demo correlation ID `25f4cde2-5251-42fb-8629-eee4fb5998a0` and the ACR Tasks build volume ID `a123076a-4e57-4df4-920b-2e0238d71c5e` are retained as synthetic example identifiers per AGENTS.md.

## Capture timeline

The lab evidence was captured by the lab's own `trigger.sh` (Phases 1-6: deployment result + per-operation MANIFEST_UNKNOWN extraction + container app baseline + revision list + ACR repository list + system log error + activity log + H1 gate emission) and `fix-and-capture.sh` (Phases 7-16: `az acr build` + `az containerapp update` + Healthy poll + post-fix container app state + revision list + ACR repository list + ACR tag list + 10 curl probes + post-fix KQL + H2 gate emission + metadata). The Phase B refactor in this commit **reuses that 2026-06-22 evidence as the canonical baseline** rather than redeploying ā€” three reasons drive that decision:

1. **Oracle Lab 21 strategy directive (2026-06-24, `ses_105f57cf7ffeKwLIxEdGFI0JmQ`)** ā€” "Pick `acr-pull-failure`. The 2026-06-22 cohort already carries the deployment-level smoking gun (`MANIFEST_UNKNOWN` in `01-deployment-operations-failed.json`), the deployment-vs-resource-state split (`provisioningState=Failed`, `latestRevisionName=null`, empty revision list, missing ACR repository), and the post-fix recovery materialization (Healthy revision, 10/10 HTTPS 200, ACR repo+tag both present, post-fix KQL silent_acceptable). Four-gate scope: cohort integrity, failure attribution, recovery materialization, in-place recovery continuity. Keep the headline scoped to 'this 2026-06-22 reproduction in Korea Central' ā€” do not generalize the deployment vs. resource-state behavior to platform-wide guarantees without re-running across regions and platform versions."
2. **Cost minimization (carry-over user constraint)** ā€” A redeploy would provision one Log Analytics workspace, one Container Apps Environment, one ACR (Basic SKU), and one Container App in `Failed` state for ~3 minutes followed by ~6 minutes of `Succeeded` state for `fix-and-capture.sh`, costing under USD $0.05 per re-run, but the existing 2026-06-22 cohort already contains every field the new four-gate predicate suite needs, the smoking-gun error message is deterministic across reproductions (the manifest lookup either succeeds or fails with `MANIFEST_UNKNOWN` ā€” there is no third state), and the post-fix Healthy revision + curl result is a binary outcome that does not benefit from re-measurement.
3. **The deployment-vs-resource-state distinction is the central evidence claim** ā€” A fresh re-run might land the per-operation error message under a different `properties.statusMessage` schema path if the resource provider evolves the deployment-operations JSON shape, and the existing cohort already documents the exact JSONPath where the smoking gun lives (`properties.statusMessage.error.message` per `01-deployment-operations-failed.json`, NOT `properties.error.message` per `01-deployment-result.json` which only carries the generic ARM "At least one resource deployment operation failed" envelope). The Phase B gates are written against the captured schema, so a future schema change would surface as a Gate 31 sub-gate (a) failure rather than a silent acceptance of a now-incorrect predicate.

## Why this is a deployment-level failure attribution lab, not a KQL-attribution lab

This lab does NOT claim "ACR pull failures emit attribution rows into `ContainerAppSystemLogs_CL`." That claim is structurally wrong for this failure mode. The other labs in this evidence pack test failure modes where the container starts and the platform emits attribution rows that a KQL query can detect (`ingress-target-port-mismatch`, `cpu-throttling`, `memory-leak-oomkilled`). ACR pull failure is different: when the manifest lookup fails during deployment, the platform never creates a revision, never starts a container, and never emits any rows into the system log table. The `ContainerAppSystemLogs_CL` table is not even materialized in the Log Analytics workspace during the failed-deployment window. The lab's headline claim is therefore scoped to the **deployment-level** signals and is capped at `[Observed]` for the single-reproduction window:

- **Gate 30 (cohort integrity, 4 sub-gates) ā€” the cohort is uncontaminated and temporally coherent.** All 24 canonical Phase A files exist (sub-gate a ā€” Strong path requires all 24; Fallback path requires >= 22 to tolerate the case where the metadata capture phase fails to write `21-cli-containerapp-ext.json` separately from `20-cli-versions.json`). The deployment timestamp, the trigger.sh start line, and the fix-and-capture.sh start line are monotonic and span <= 30 minutes (sub-gate b ā€” Strong path requires monotonic AND <= 30 min; Fallback path relaxes the span to <= 60 min but still requires monotonic; observed span 2026-06-22T13:20:46Z ā†’ 13:26:42Z is **355.9 seconds**, well under both thresholds; the temporal check uses runtime-captured `tee` output lines, NOT the backfilled `06-h1-gate.json.utc_captured` field, per the Honest disclosure note below). There are zero unexpected non-junk extras under `evidence/` (sub-gate c ā€” Strong path requires exact match of the 24 canonical + 4 Phase B + README; Fallback path tolerates editor/OS junk like `.swp`, `.bak`, `.tmp`, `.DS_Store` but still requires zero missing canonical files ā€” the `unexpected_non_junk_extras` predicate per Lab 19 P0 lesson and the Lab 20 PR #279 P0 fallback-integrity-preservation directive). This README cross-references all 4 Phase B gate JSON filenames (sub-gate d ā€” a reviewer reading this README can locate every gate output).

- **Gate 31 (failure attribution, 5 sub-gates) ā€” the failed-deployment cohort is fully attributable to manifest resolution failure.** The deployment terminated `Failed` AND at least one per-operation record in `01-deployment-operations-failed.json` contains `MANIFEST_UNKNOWN` in `statusMessage.error.message` (sub-gate a ā€” single-path per Oracle Q2; the smoking gun lives at exactly one canonical JSONPath, `01-deployment-operations-failed.json[*].properties.statusMessage.error.message`, NOT at `01-deployment-result.json.properties.error.message` which only carries the generic ARM "At least one resource deployment operation failed" envelope whose `details[*].details` is `null` at the leaf). The Container App resource was created in `provisioningState=Failed` (sub-gate b ā€” single-path), `latestRevisionName` is `null` (sub-gate c ā€” single-path), the revision list is the empty array (sub-gate d ā€” single-path), and the ACR repository list does not contain `labacr` (sub-gate e ā€” single-path, record-scoped membership per Lab 15 lesson 34: parsed JSON array membership, not greppy substring matching). **The gate JSON carries an explicit `cohort_binding_note` field** stating that sub-gates b-e are absence-style predicates that are interpretable as manifest-failure evidence ONLY in conjunction with sub-gate a ā€” a reader must not read "revision list empty" or "ACR repo absent" as evidence of this specific failure mode without also seeing the Failed deployment with `MANIFEST_UNKNOWN`. This bound-cohort framing enforces Oracle's "Bound every absence predicate to the failed-deployment cohort" directive.

- **Gate 32 (recovery materialization, 4 sub-gates) ā€” `az acr build` + `az containerapp update` materialized a Healthy serving revision.** `08-containerapp-update-result.json` shows `provisioningState=Succeeded` AND `latestRevisionName=ca-labacr-rnnljl--mvlbg1j` (sub-gate a ā€” single-path, BOTH-not-OR per Lab 19 P0 lesson). `10-revisions-list-after-fix.json` contains at least one record with `healthState=Healthy` AND `trafficWeight=100` (sub-gate b ā€” single-path, record-scoped iteration over the array, BOTH-not-OR ā€” observed: 1 record, Healthy, traffic 100, RunningAtMaxScale, 1 replica). `12-curl-after-fix.requests_ok == 10` (sub-gate c ā€” Strong path requires `== 10`; Fallback path requires `>= 8` to tolerate cold-start variance on the first few probes; **this is the only Gate 32 sub-gate with dual paths** per Oracle Q1 directive because the curl floor is the only place where two genuinely equivalent manifestations of the recovery-success claim exist ā€” "all 10 OK" and "8-9 OK with cold-start variance" both prove the recovery succeeded). `11-acr-repository-list-after-fix.json` contains `labacr` AND `11-acr-repository-show-tags-after-fix.json` contains `v1` (sub-gate d ā€” single-path, BOTH-not-OR, record-scoped membership).

- **Gate 33 (in-place recovery continuity, 3 sub-gates) ā€” the recovery is in-place: same app, same registry, image tag is the only delta.** Baseline `02-containerapp-show-baseline.name == ca-labacr-rnnljl` AND after-fix `09-containerapp-show-after-fix.name == ca-labacr-rnnljl` (sub-gate a ā€” single-path). The baseline cohort (`provisioningState=Failed`, `latestRevisionName=null`, `revs=[]`) transitions to the after-fix cohort (`provisioningState=Succeeded`, `latestRevisionName` non-empty, >= 1 Healthy revision) (sub-gate b ā€” single-path, BOTH-not-OR across cohort fields). The baseline `.image` ends with `:does-not-exist`, the after-fix `.image` ends with `:v1`, AND the registry prefix (everything before the first `/`) is identical across both images (sub-gate c ā€” single-path, record-scoped on `.image` field; the registry identity is explicit in the parsed JSON field so the same-registry proof is sound per Oracle Q1; **timestamp / no-recreation claims dropped per Oracle Q1 as not soundly evidenced by this cohort** ā€” the cohort cannot prove the underlying resource was not torn down and recreated with the same name in the same span, so the gate does not assert that).

The deployment-level **claim ceiling** is the entire reason the lab exists in its current form. A claim like "ACR pull failures always produce `MANIFEST_UNKNOWN` and never any other ImagePullBackOff variant" would require multiple reproductions across image-not-found, registry-unreachable, credential-expired, and `denied` (registry exists but blocks access) scenarios. The lab instead proves a much narrower thing ā€” **at this 2026-06-22 reproduction in Korea Central, the specific failure mode of "image tag not present in referenced ACR" produced a deployment-level signature with `MANIFEST_UNKNOWN` and the recovery via `az acr build` + `az containerapp update` was in-place** ā€” and explicitly hands the operator the conclusion: the cheapest live-incident diagnostic for this failure mode is the combination of `az deployment operation group list --query "[?properties.provisioningState=='Failed'].properties.statusMessage"` (per-operation error) plus `az containerapp revision list` returning `[]` (no revision materialized); either alone is suggestive, but the two together are the smoking gun. A managed-identity-based `AcrPull`-role-missing variant of this failure mode is documented separately in [`docs/troubleshooting/playbooks/startup-and-provisioning/image-pull-failure.md`](../../../docs/troubleshooting/playbooks/startup-and-provisioning/image-pull-failure.md) and is not part of this scripted reproduction.

## Honest disclosure ā€” empirical platform behavior captured during this run

The following observations surfaced during the 2026-06-22 historical run and are documented here so a future operator (or Oracle reviewer) can interpret the raw evidence without re-discovering each gap:

- **`06-h1-gate.json` timestamp was backfilled from the trigger run window.** The `utc_captured` field in `evidence/06-h1-gate.json` (`2026-06-22T13:34:00Z`) was backfilled from the trigger run window after the original Bash quoting bug was identified ā€” the original `trigger.sh` run wrote the literal string `$(date -u +%Y-%m-%dT%H:%M:%SZ)` into the field because the Python heredoc that emits the gate JSON was single-quoted (`<<'PYEOF'`), which disables `$(...)` substitution. The fix (resolve `UTC_CAPTURED` in Bash before the heredoc and read it via `os.environ['UTC_CAPTURED']` inside Python) is committed in `trigger.sh`, but the resource group `rg-aca-lab-acr` had already been deleted by `cleanup.sh` and could not be regenerated. The backfilled timestamp is bounded by the run log: `evidence/00-trigger-run.txt:1` shows `trigger.sh starting at 2026-06-22T13:25:49Z`, and the same file's mtime is `2026-06-22T13:34:11Z` (script termination). Per Oracle Q4 directive, this caveat is **also restated in the lab-level README** ([`labs/acr-pull-failure/README.md`](../README.md) ā€” see the "06-h1-gate.json timestamp backfill" admonition) so future readers see it both at the lab landing and at the evidence-pack entry. **Gate 30 sub-gate b uses runtime-captured `tee` output lines, NOT the backfilled `06-h1-gate.json.utc_captured` field, to evaluate temporal coherence** ā€” this is explicitly noted in the gate JSON's `sub_gate_b_predicate` text.

- **The legacy `00-verify-run.txt` filename is preserved for schema stability.** Phase A captured the output of the script that was named `verify.sh` at the time (2026-06-22). The Phase B refactor in this commit **renamed that script to `fix-and-capture.sh`** (so Phase A operators still have the same recovery orchestration under a more semantically accurate name ā€” the script actually FIXES the failure rather than just observing it) and reassigned the `verify.sh` name to the new gate-emission processor. The captured log file is still named `00-verify-run.txt` to preserve the canonical-evidence schema across the rename ā€” re-naming the file would invalidate every existing reference in the playbook, the lab guide, and prior commits. The script header inside `fix-and-capture.sh` documents this decision; the file content captured on 2026-06-22 still echoes `verify.sh starting at ...` because that is what the script was called at the time of capture, but future captures will echo `fix-and-capture.sh starting at ...` per the updated echo strings.

- **The lab uses ACR admin user, not managed identity.** The point of this lab is to prove that the image tag is the controlling variable, not the registry credential mechanism. Using ACR admin user (enabled in Bicep `infra/main.bicep`, password stored in the `registry-password` secret) keeps the credential path uniform across the failing run and the recovered run; the v1 image pull succeeds with the same admin credential that was already present during the bad-tag run, so the failure cannot be attributed to a credential issue. A managed-identity variant of this failure mode (with the `AcrPull` role missing) is documented separately in the playbook ([`docs/troubleshooting/playbooks/startup-and-provisioning/image-pull-failure.md`](../../../docs/troubleshooting/playbooks/startup-and-provisioning/image-pull-failure.md)) and is not part of this scripted reproduction.

- **The container image is a custom Flask + Gunicorn build, not a public placeholder.** This lab is the entry point of the ACR pull-failure scenario in the playbook chain. Using a custom image (rather than `mcr.microsoft.com/k8se/quickstart` or `containerapps-helloworld`) ensures the lab actually exercises the ACR pull path; a public Microsoft-hosted image would never trigger the ACR-specific failure mode because there is no ACR pull involved. The workload itself is intentionally minimal (Flask + Gunicorn on `:8000`, exposing `GET /` and `GET /health`) so the focus stays on the registry pull behavior, not on the application.

- **`az acr build` runs on a Microsoft-hosted ACR Tasks worker, not on the local Docker daemon.** Phase 8 of `fix-and-capture.sh` calls `az acr build --registry acrlabacrrnnljl --image labacr:v1 ./workload`, which runs the Dockerfile build inside an ACR Tasks worker (build volume ID `a123076a-4e57-4df4-920b-2e0238d71c5e` captured in `07-acr-build-result.json` is a synthetic example identifier per AGENTS.md, not a real account-scoped resource ID). The lab therefore does not depend on a local Docker daemon, Docker login state, or `docker push` upload bandwidth ā€” `az acr build` is more reproducible across operator environments because it does not depend on the presence of a local container runtime. The recovery is identical in observable behavior to a local `docker build` + `docker push` (both produce a `labacr:v1` tag in ACR with the same image contents).

- **`23-deployment-outputs.json` is captured intentionally despite `properties.outputs` being `null`.** The Azure Resource Manager platform only populates `properties.outputs` on `Succeeded` deployments. In the `Failed` state, the output blob is `null`, and the file captures that null value plus the deployment envelope (correlationId, timestamp, mode, parameters). The resource names the scripts use (`APP_NAME`, `ACR_NAME`, `ACR_LOGIN_SERVER`, `WORKSPACE_CUSTOMER_ID`) are derived in Quick Start step 3 of [`labs/acr-pull-failure/README.md`](../README.md) from `az containerapp list`, `az acr list`, and `az monitor log-analytics workspace list` (one-row queries against the dedicated RG) rather than from deployment outputs. Capturing the `null` outputs explicitly documents this platform behavior so a future operator does not waste cycles trying to read outputs from a Failed deployment.

- **The post-fix KQL query observed 0 rows and is classified `silent_acceptable_post_fix` (informational only).** Phase 15 of `fix-and-capture.sh` issues a post-fix KQL query against `ContainerAppSystemLogs_CL` and captured the result into `13-kql-after-fix.json` (`row_count: 0`, `classification: silent_acceptable_post_fix`). Per Oracle Q5 directive, **this is informational only and is NOT promoted to a gate** ā€” the post-fix silence is consistent with platform-side ingestion lag (Log Analytics commonly takes 2-5 minutes to surface fresh rows from a freshly-materialized custom table) and the H2 PASS verdict is anchored to the deterministic Healthy + traffic + curl signals (Gate 32 sub-gates a/b/c/d), not to the KQL row count. A future operator who suspects the table is genuinely not materializing post-fix can extend the wait window in Phase 15 and re-run; the current 2-minute wait window is calibrated for the lab's wall-clock budget, not for guaranteed ingestion completion.

- **The correlation ID `25f4cde2-5251-42fb-8629-eee4fb5998a0` and the ACR Tasks build volume ID `a123076a-4e57-4df4-920b-2e0238d71c5e` are retained as synthetic example identifiers per AGENTS.md.** These GUIDs are NOT real subscription, tenant, object, or resource identifiers ā€” they are per-deployment correlation tokens and per-build artifact volumes that surface in the captured CLI output. AGENTS.md explicitly permits "Demo correlation IDs" and "Example request IDs in logs" to be retained without zero-GUID replacement; replacing them would strip the structural shape that aids operator pattern-matching when reading the raw JSON.

## Reproducibility

To reproduce this evidence pack against a fresh Azure subscription:

```bash
export AZ_SUBSCRIPTION="<your-subscription-id>"
export RG="rg-aca-lab-acr"
export LOCATION="koreacentral"

az group create --name "$RG" --location "$LOCATION" --subscription "$AZ_SUBSCRIPTION"

# Deploy the lab (RG + Log Analytics + Container Apps env + ACR + 1 app with
# image:does-not-exist). The deployment is EXPECTED to fail with MANIFEST_UNKNOWN;
# `|| true` lets the script proceed past the failure so the failed-deployment
# state can be captured.
az deployment group create \
    --subscription "$AZ_SUBSCRIPTION" \
    --resource-group "$RG" \
    --name main \
    --template-file labs/acr-pull-failure/infra/main.bicep \
    --parameters baseName="labacr" || true

# Phase A ā€” deployment-level failure attribution capture (~9 minutes, < $0.05 cost).
# trigger.sh:           Phases 1-6 (deployment result + per-op MANIFEST_UNKNOWN +
#                       baseline app state + revisions [] + ACR repos [] + system
#                       log error + activity log + H1 gate emission).
# fix-and-capture.sh:   Phases 7-16 (az acr build + az containerapp update + Healthy
#                       poll + post-fix app state + revisions + ACR repos + curl 10
#                       probes + post-fix KQL + H2 gate emission).
bash labs/acr-pull-failure/trigger.sh
bash labs/acr-pull-failure/fix-and-capture.sh

# Phase B ā€” pure file processor (NO Azure calls) ā€” emits gates 30..33 against the
# canonical cohort in evidence/. Can be re-run by reviewers without re-deploying.
bash labs/acr-pull-failure/verify.sh

# Async cleanup (--no-wait, costs stop accruing immediately).
bash labs/acr-pull-failure/cleanup.sh
```

Expected runtime: **~12 minutes total** (~3 min Bicep deploy attempt, ~2 min `trigger.sh` capture, ~6 min `fix-and-capture.sh` including the 5-min Healthy-poll budget, ~1 second `verify.sh` gate emission, immediate cleanup queue). Estimated cost: **< USD $0.05** (one Log Analytics workspace on PerGB ingestion, one Consumption-plan Container Apps Environment, one ACR Basic SKU prorated to ~$0.001 over the lab window, one Container App in `Failed` state for most of the lab then `Succeeded` post-recovery with `minReplicas: 1`, `maxReplicas: 1`, `0.5 vCPU`, `1 Gi` memory for ~2 minutes of post-fix probes). The ACR Basic SKU is the dominant cost driver at approximately USD $0.167/day prorated to the hour; the `az acr build` Tasks worker is billed per-build-minute (the Flask + Gunicorn build typically completes in under 2 minutes for well under USD $0.01).

The Phase B `verify.sh` is a pure file processor ā€” it reads `evidence/*.json` and `evidence/*.txt` from disk and emits `evidence/{30,31,32,33}-*-gate.json`. It does NOT call Azure, which is why the resource group can be deleted (via `cleanup.sh`) before `verify.sh` finishes the gate-emission phases. The pre-Phase-B `verify.sh` was the Phase A live-Azure recovery + capture script (Phases 7-16); the Phase B refactor in this commit **renamed that script to `fix-and-capture.sh`** (so Phase A operators still have the same recovery orchestration under a more semantically accurate name ā€” the script actually FIXES the failure rather than just observing it) and reassigned the `verify.sh` name to the new gate-emission processor. `trigger.sh` performs Phases 1-6 (deployment-level capture) and is unchanged.
