# Evidence Pack Provenance

This directory contains the raw evidence artifacts for the `revision-history-limit` lab. The lab is a **bounded-observation** test of the Azure Container Apps preview setting `maxInactiveRevisions` (CLI alias `--revision-history-limit`) — a single Container App is provisioned with `maxInactiveRevisions: 2`, then driven through a 10-update env-var-only burst, and the live revision list is captured at three fixed offsets after the burst (**t+0, t+5m, t+15m**). The lab does NOT claim a generalized pruning SLA; it captures one 15-minute observation window from one reproduction and tests only that (a) the configured value persists across the window and (b) pruning was not prompt within that window. The cohort was captured in a single live-Azure window on **2026-06-22** against RG `rg-aca-lab-revhist` (Korea Central, azure-cli 2.79.0, containerapp extension 1.3.0b4 preview). All files are PII-scrubbed (subscription/tenant GUIDs replaced with the zero-GUID placeholder, employee aliases replaced with `demouser`, employee emails replaced with `user@example.com`).

## Capture timeline

The lab evidence was captured by the lab's own `trigger.sh` (Phases 1-4: config readback, initial revisions, burst 10 updates, t+0 capture) and `capture-window.sh` (Phases 5-7: t+5m wait + capture, t+15m wait + capture, t+15m config readback). The Phase B refactor in this commit **reuses that 2026-06-22 evidence as the canonical baseline** rather than redeploying — three reasons drive that decision:

1. **Oracle Lab 20 directive (2026-06-24, `ses_1069d765dffezi1iYs4KKtHH6r`)** — "Pick `revision-history-limit`. Cleanest pure-file cohort with an explicit anchor in `burst-completed-iso.txt` (epoch 1782115306, ISO 2026-06-22T08:01:46Z). Four-gate scope: cohort integrity, config persistence, burst materialization, bounded-window non-pruning. Keep the headline to 'pruning was not prompt within this 15-minute observation window in this reproduction.' Do not generalize to platform-wide timing guarantees."
2. **Cost minimization (carry-over user constraint)** — A redeploy would provision one Log Analytics workspace, one Container Apps Environment, and one Container App for ~20 minutes of wall-clock, costing under USD $0.05 per re-run, but the existing 2026-06-22 cohort already contains every field the new four-gate predicate suite needs, and the bounded-observation hypothesis is intentionally re-run-tolerant (the gates do not encode the exact t+0 / t+5m / t+15m counts as hard equalities — they encode floors and monotonicity).
3. **The pruning cadence is undocumented and not part of any published SLA** — `maxInactiveRevisions` is a preview setting and the pruning interval is not specified anywhere in Microsoft Learn or the resource provider schema. A fresh re-run might observe pruning at minute 12 or minute 14 instead of "still not pruned at minute 15," and a Strong-path-only gate that hardcoded `inactive_count == 10` at t+15m would falsely flag that legitimate behavior change as a regression. The Phase B gates are therefore written with a Fallback path keyed off the configured limit (`> 2`) plus a Strong path keyed off the observed floor (`≥ 8`), and they pass if either path is true.

## Why this is a bounded-observation lab, not a pruning-SLA lab

This lab does NOT claim "Container Apps prunes inactive revisions within N minutes of exceeding `maxInactiveRevisions`." That claim is too strong for what one 15-minute observation window can prove. The platform exposes the *target* via `properties.configuration.maxInactiveRevisions`, but the reconciliation interval that drives pruning is **not documented** anywhere in Microsoft Learn or the Azure Resource Manager schema for `Microsoft.App/containerApps`. The lab's headline claim is intentionally narrow and capped at `[Observed]` for the single-reproduction window only:

- **Gate 20 (cohort integrity, 4 sub-gates) — the cohort is uncontaminated.** All 14 canonical Phase A files exist (sub-gate a — Strong path requires all 14; Fallback path requires ≥12 to tolerate the case where a re-runner forgets to capture `08-cli-containerapp-ext.json` separately from `07-cli-versions.json`). The anchor files `burst-completed-epoch.txt` (`1782115306`) and `burst-completed-iso.txt` (`2026-06-22T08:01:46Z`) are reciprocal — converting the epoch with `datetime.fromtimestamp(epoch, tz=UTC)` produces the same ISO string in the ISO file (sub-gate b — Strong path requires exact reciprocal match between the epoch-derived ISO and the ISO file content, proving the anchor pair was emitted atomically by the same `trigger.sh` run, not stitched together from different runs; Fallback path enforces an intent-preserving agreement check that requires the parsed epoch and the parsed ISO timestamp to be within ≤2 s drift per Oracle PR #279 P0-2 directive — the previous fallback of "both files parseable" was too permissive because it would have accepted an anchor pair stitched together from different runs with hours or days of drift, weakening the documented §4b contract; the emitted `drift_seconds` and `drift_tolerance_seconds` fields in the gate JSON record the measured drift for operator forensics). There are zero unexpected non-junk extras under `evidence/` (sub-gate c — Strong path requires exact match of the 14 canonical + 4 Phase B + README; Fallback path tolerates editor/OS junk like `.swp`, `.bak`, `.tmp`, `.DS_Store` but still requires zero missing canonical files — the `unexpected_non_junk_extras` predicate per Lab 19 P0 lesson). This README cross-references all 4 Phase B gate JSON filenames (sub-gate d — a reviewer reading this README can locate every gate output).

- **Gate 21 (config persistence, 2 sub-gates) — the setting was honored as a target across the window.** Pre-burst config readback (`01-app-config-before.json`) shows `{activeRevisionsMode: "Single", maxInactiveRevisions: 2}` (sub-gate a — Strong AND Fallback paths BOTH enforce the same both-fields-match predicate per Oracle PR #279 P0-3 directive; the dual-path shape exists for JSON schema parity, NOT for semantic relaxation; the earlier fallback of "`maxInactiveRevisions` is a finite non-negative integer" was too permissive because it would have accepted the platform default of `100`, falsely flagging an un-set Bicep value as PASS). Post-window config readback (`06-app-config-t15m.json`) shows the same `{Single, 2}` AND is **byte-identical** (67 B == 67 B) to the pre-burst capture (sub-gate b — Strong path proves there was no mid-window mutation, e.g., no rogue `az containerapp update --max-inactive-revisions=N` ran during the 15-minute window; Fallback path only requires both fields match without byte-identity).

- **Gate 22 (burst materialization, 3 sub-gates) — the 10 env-var-only updates actually created 10 distinct new revisions.** `02-revisions-initial.json` has exactly 1 revision before the burst (sub-gate a — Strong AND Fallback paths BOTH enforce `initial_count == 1` per Oracle PR #279 P0-4 directive — weakening the fallback to `≥ 1` would silently break sub-gates b/c whose +10 burst-suffix interpretation assumes a 1-revision starting point; the dual-path shape exists for JSON schema parity, NOT for semantic relaxation). `03-revisions-t0.json` has exactly 11 revisions after the burst (sub-gate b — Strong path requires `== 11` per Oracle directive for the ideal burst; Fallback path requires `≥ 8` to tolerate the case where 1-3 of the env-var updates landed on the same template hash and did not materialize as new revisions). All 11 revision names are distinct (sub-gate c — Strong path requires `== 11` distinct names; Fallback path requires `≥ 8` distinct, falsifying any "the env-var update no-op'd and the same revision was reused" theory). The 10 burst revisions are `--0000001` through `--0000010` plus the original `--wdmdnoq`.

- **Gate 23 (bounded-window non-pruning, 3 sub-gates) — pruning was not prompt within this 15-minute window in this reproduction.** At t+5m the inactive revision count is 10, which is `> 2` (the configured limit) and `≥ 8` (the strong floor) — sub-gate a passes on both paths. At t+15m the inactive revision count is still 10, same numbers — sub-gate b passes on both paths. **BOTH endpoints are above the configured target AND the count is monotonic non-decreasing** between t+5m and t+15m (sub-gate c — Strong path requires both `≥ 8` AND monotonic per Lab 19 P0 BOTH-not-OR lesson; Fallback path requires both `> 2` AND monotonic). The monotonic-non-decrease check explicitly allows the inactive count to *grow* between t+5m and t+15m (which would be unusual — no new bursts ran during the window — but is not a falsification of "pruning was not prompt"), but does NOT allow the count to *shrink* below itself (which would be ambiguous: pruning happened mid-window but the gate cannot distinguish "pruned to 3, still > 2" from "pruning is prompt").

The bounded-observation **claim ceiling** is the entire reason the lab exists in its current form. A claim like "Container Apps prunes inactive revisions on a 30-minute reconciliation interval" would require multiple reproductions across regions, time-of-day, and platform versions, plus a comparison against a documented platform SLA that does not exist. The lab instead proves a much narrower thing — **at minute 15 of this specific reproduction, pruning had not yet occurred** — and explicitly hands the operator the conclusion: do not depend on `maxInactiveRevisions` as a short-window cleanup mechanism; use explicit `az containerapp revision deactivate` calls when you need deterministic cleanup with a known timing guarantee.

## Honest disclosure — empirical platform behavior captured during this run

The following observations surfaced during the 2026-06-22 historical run and are documented here so a future operator (or Oracle reviewer) can interpret the raw evidence without re-discovering each gap:

- **`maxInactiveRevisions` is a preview feature with no documented pruning SLA.** Microsoft Learn ([Container Apps revisions](https://learn.microsoft.com/en-us/azure/container-apps/revisions)) documents the setting but does not specify how often the platform reconciles the inactive revision count against the configured limit. The default cap is 100 inactive revisions (not 10 or 2). This lab sets the limit to 2 explicitly to make the gap between configured target and observed live count visible after only 10 updates instead of 100+. The Strong-path threshold of 8 in Gate 23 (`inactive ≥ 8`) is calibrated for the "burst of 10" cohort; a re-run with a smaller burst would need to adjust this floor.

- **The 15-minute window is empirically chosen, not platform-documented.** `capture-window.sh` Phase 6 sleeps 597 seconds (~10 minutes) after the Phase 5 t+5m capture, landing at t+15m relative to the burst-end anchor. The choice of 15 minutes balances "long enough for any reasonable reconciliation interval shorter than that to fire" against "short enough to keep the lab under 20 minutes of wall clock and well under $0.05 cost." A future operator who suspects a longer reconciliation interval can extend the window by editing the sleep math in `capture-window.sh` Phase 6 (and adding Phase 8 captures at t+30m, t+60m, etc.) — but doing so changes the cohort, requires a new burst-anchor, and produces a different evidence pack.

- **The legacy `00-verify-run.txt` filename is preserved for schema stability.** Phase A captured the output of the script that was named `verify.sh` at the time (2026-06-22). The Phase B refactor in this commit **renamed that script to `capture-window.sh`** (so Phase A operators still have the same readiness gate under a more semantically accurate name) and reassigned the `verify.sh` name to the new gate-emission processor. The captured log file is still named `00-verify-run.txt` to preserve the canonical-evidence schema across the rename — re-naming the file would invalidate every existing reference in the playbook, the lab guide, and prior commits. The script header inside `capture-window.sh` documents this decision; the file content captured on 2026-06-22 still echoes "verify.sh started" because that is what the script was called at the time of capture, but future captures will echo "capture-window.sh started" per the updated echo strings.

- **`activeRevisionsMode: Single` keeps the experimental variable isolated.** The lab uses `Single` mode so each `az containerapp update` deactivates the previous active revision and promotes the new one. This produces a clean 1-active / N-inactive split that the Gate 23 inactive-count check can read deterministically. If the lab were re-run with `activeRevisionsMode: Multiple`, the burst-end inactive count would depend on traffic-weight policy and the Gate 23 floor would need to be reinterpreted.

- **The burst nonce `20260622075836` is preserved verbatim in the captured logs and is not PII.** `00-trigger-run.txt` lines 26 and 27 record `Burst nonce: 20260622075836` and `Update 1/10 (REV=20260622075836-1)`. This nonce is a per-run UTC timestamp generated by `trigger.sh` to ensure each env-var-only update is unique (otherwise consecutive updates with the same `REV=` value would no-op and would NOT create new revisions). The nonce is purely a lab-internal correlation token and carries no Azure account information.

- **The container image is the public placeholder `mcr.microsoft.com/azuredocs/containerapps-helloworld:latest` for the full lab.** No ACR is provisioned, no `az acr build` runs, no custom image is pushed. This keeps the experimental variable (the inactive-revision retention limit) isolated from any image-rebuild side effects, makes the lab redeployable in any region that supports Container Apps without a private registry, and keeps the cost near zero. The 11 revisions in `03-revisions-t0.json` all share the same image — only the `REV` env var differs between them.

- **`scale.sh` and `sample.sh` from Lab 19 do NOT exist in this lab.** This lab uses single-app, single-window capture instead of a multi-profile differential. The orchestration is the much simpler `trigger.sh` (4 phases, ~3.5 min) → `capture-window.sh` (3 phases, ~15 min). The Phase B `verify.sh` is the gate-emission processor (no Azure calls).

- **The pruning interval is undocumented and may vary across regions, time-of-day, and platform versions.** The Gate 23 claim is explicitly scoped to "this 15-minute observation window in this reproduction" precisely because the cohort cannot prove a generalized pruning cadence. A re-run on a different day, in a different region, or after a platform update may observe pruning at minute 8 or never within minute 60 — both would be consistent with "the setting is honored as a target, not as a short-window SLA." The Fallback paths in Gates 22 and 23 are designed to keep the lab's pass/fail logic stable across reasonable re-run variability.

## Reproducibility

To reproduce this evidence pack against a fresh Azure subscription:

```bash
export AZ_SUBSCRIPTION="<your-subscription-id>"
export RG="rg-aca-lab-revhist"
export LOCATION="koreacentral"

az group create --name "$RG" --location "$LOCATION" --subscription "$AZ_SUBSCRIPTION"

# Deploy the lab (RG + Log Analytics + Container Apps env + 1 app with
# maxInactiveRevisions=2, helloworld image, minReplicas=0).
az deployment group create \
    --subscription "$AZ_SUBSCRIPTION" \
    --resource-group "$RG" \
    --name main \
    --template-file labs/revision-history-limit/infra/main.bicep \
    --parameters baseName="revhist"

# Phase A — bounded-observation capture (~18 minutes, < $0.05 cost).
# trigger.sh: Phases 1-4 (config read, initial revisions, burst 10 updates, t+0 capture).
# capture-window.sh: Phases 5-7 (wait + capture at t+5m, wait + capture at t+15m, config readback at t+15m).
bash labs/revision-history-limit/trigger.sh
bash labs/revision-history-limit/capture-window.sh

# Phase B — pure file processor (NO Azure calls) — emits gates 20..23
# against the canonical cohort in evidence/. Can be re-run by reviewers
# without re-deploying.
bash labs/revision-history-limit/verify.sh

# Async cleanup (--no-wait, costs stop accruing immediately).
bash labs/revision-history-limit/cleanup.sh
```

Expected runtime: **~20 minutes total** (~3 min Bicep deploy, ~3.5 min `trigger.sh` burst, ~15 min `capture-window.sh` wait + sample, ~1 second `verify.sh` gate emission, immediate cleanup queue). Estimated cost: **< USD $0.05** (one Log Analytics workspace on PerGB ingestion, one Consumption-plan Container Apps Environment, one Container App with `minReplicas: 0` that scales to zero between updates, no ACR, no Application Insights, no public IP, no private endpoint, Korea Central). The Log Analytics workspace is the dominant cost driver but only ingests the ~10 KB of platform logs produced during the 18-minute window.

The Phase B `verify.sh` is a pure file processor — it reads `evidence/*.json` and `evidence/*.txt` from disk and emits `evidence/{20,21,22,23}-*-gate.json`. It does NOT call Azure, which is why the resource group can be deleted (via `cleanup.sh`) before `verify.sh` finishes the gate-emission phases. The pre-Phase-B `verify.sh` was the Phase A live-Azure t+5m / t+15m capture script; the Phase B refactor in this commit **renamed that script to `capture-window.sh`** (so Phase A operators still have the same capture orchestration under a more semantically accurate name) and reassigned the `verify.sh` name to the new gate-emission processor. `trigger.sh` performs Phases 1-4 (config + initial revisions + burst + t+0 capture) and is unchanged.

## CLI versions and platform context

The captures in this pack were produced with **azure-cli 2.79.0** and the **`containerapp` extension 1.3.0b4 (preview)** on Korea Central. The lab REQUIRES the `containerapp` extension at preview because `--max-inactive-revisions` is a preview-only CLI flag at this extension version; install or upgrade with `az extension add --name containerapp --upgrade --allow-preview true`. The only Azure-side resources are the Log Analytics workspace (`log-revhist-t3t3np`, PerGB pricing), the Consumption-plan Container Apps Environment (`cae-revhist-t3t3np`), and the Container App (`ca-revhist-t3t3np`) running the public placeholder image `mcr.microsoft.com/azuredocs/containerapps-helloworld:latest`. The lab does not provision an ACR; the placeholder image is pulled directly from the public MCR registry on every revision activation. The `maxInactiveRevisions` value is set in **Bicep only** (`infra/main.bicep`) and is never mutated via the CLI during the lab, which is why the byte-identity check in Gate 21 sub-gate (b) is meaningful — any mid-window CLI mutation would change the JSON serialization order or whitespace and break the byte-identity, even if the integer value were the same.
