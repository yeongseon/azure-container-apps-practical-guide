# Evidence pack — `cpu-throttling` lab

This directory carries the committed canonical evidence cohort for the `cpu-throttling` lab, anchored on the live-Azure deployment in `koreacentral` on 2026-06-22. The cohort consists of 15 canonical Phase A files (2 script logs + 2 before-fix captures + 1 baseline load-test result + 1 baseline metric snapshot + 1 update result + 2 after-fix captures + 1 post-fix load-test result + 1 post-fix metric snapshot + 4 environment captures), plus 4 Phase B falsification-gate JSONs emitted by `labs/cpu-throttling/verify.sh`, plus this README. The cohort contains no Azure account PII to scrub: the captured `az` queries deliberately project only resource-state fields (`cpu`, `memory`, `replicas`, `revisionName`, `trafficWeight`, `runningState`, `latencyMs`, etc.), no subscription / tenant / object GUIDs, no employee aliases, no email addresses, and no secrets are present in any canonical file. The Container Apps environment domain segment `graymeadow-18a4aa16` that appears inside the FQDN is the platform-assigned random subdomain for this environment and is not tied to a customer-specific identifier; it is preserved as-is across the cohort for reproducibility.

## Capture timeline

Three carry-over framings to make the evidence pack readable:

1. **The workload is `python:3.12-slim` running a deterministic CPU-burn HTTP server, NOT the standard `mcr.microsoft.com/azuredocs/containerapps-helloworld:latest`.** Helloworld returns immediately without measurable CPU work, so any latency difference between cpu=0.25 and cpu=1.0 on helloworld is dominated by cold-start (~6 s spike on first request), image-pull (~1-2 s), and TLS/TCP setup (~50-100 ms) rather than steady-state CPU throttling. The inline `python:3.12-slim` server gives each GET a deterministic ~40 ms of CPU work on a full vCPU (80 SHA-256 hashes over a 200 KiB buffer), which scales to ~160 ms minimum under Linux CFS throttling at 0.25 vCPU. The signal is large enough to dwarf network jitter and isolated from cold-start because the lab warms up the replica with 5 discarded GETs before each measurement. See the lab `README.md` "Why this workload (and not helloworld)" section for the full carry-over.
2. **The lab pins `minReplicas: 1` AND `maxReplicas: 1` so scale-out cannot mask per-replica CPU pressure.** The hypothesis under test is specifically about CPU throttling at the replica level. If `maxReplicas` were higher, KEDA HTTP scale-out could spin up a second replica during the load test and the observed latency would reflect both per-replica throttling AND inter-replica request distribution. Pinning to a single replica isolates the per-replica CPU envelope as the only experimental variable on the workload axis. A separate lab (`replica-load-imbalance`) covers scale-out behavior.
3. **The single-variable claim is bounded to the shared-fields surface, not to pod reuse, image-digest identity, or a general activation-time guarantee.** The before-fix snapshot (`01-app-config-before.json`) and the after-fix snapshot (`06-app-config-after.json`) share four keys (`cpu`, `memory`, `minReplicas`, `maxReplicas`); `activeRevisionsMode` appears only in `01` and `latestRevisionName` appears only in `06` due to a documented schema asymmetry in the Phase A `az containerapp show --query` projections. On the shared-keys surface the diff set is exactly `{cpu, memory}` with `minReplicas` and `maxReplicas` unchanged at `1`. Gate 17 anchors on this shared-keys diff, explicitly documents the asymmetry as out-of-scope for the diff predicate, and explicitly drops three claims the cohort does NOT prove: (1) the underlying Kubernetes pod was reused — sub-gate (b) shows a NEW revision was created and the old revision is in `runningState==Deprovisioning`; (2) the container image is byte-identical — this is inferred from the Phase A `az containerapp update --cpu --memory` call (no `--image`, `--command`, or `--args` flag) but the cohort does not capture image digests; (3) Azure Container Apps generally activates the new revision within 15 seconds — the cohort captured one 15-second observation (`08:50:45Z runningState=Activating` → `08:51:00Z runningState=RunningAtMaxScale`), not a multi-run timing bound.

## Why this is a CPU-throttling falsification lab, not a metric-attribution lab

The `appinsights-connection-string-missing` lab uses a silent-observability-gap failure model (the container starts, HTTP requests return 200, but `AppRequests` is empty because the SDK is not wired). This lab is the opposite: telemetry IS materialized, all 100 + 100 requests return HTTP 200 across both runs, and the failure mode is a quantitative latency-tail signature at the constrained CPU envelope. The primary signal is the 100/20 load-test result captured in `03-loadtest-cpu025.json` and `08-loadtest-cpu1.json` — not the platform `UsageNanoCores` metric, which carries a documented timing limitation under short-duration load (see "Honest disclosure"). The four Phase B gates encode four orthogonal claims that together prove the CPU-throttling falsification:

- **Gate 14 — `cpu_throttling_h_cohort_integrity_evidence_cohort_is_internally_consistent_and_temporally_coherent`** (4 sub-gates). Confirms (a) all 15 canonical Phase A files exist on disk — Strong path requires exactly 15, Fallback path requires `>= 13` present AND the five hypothesis-gate inputs (`01-app-config-before.json`, `03-loadtest-cpu025.json`, `06-app-config-after.json`, `07-revisions-after.json`, `08-loadtest-cpu1.json`) all present; (b) temporal coherence — the baseline `started_utc` field of `03-loadtest-cpu025.json` (`2026-06-22T08:49:38Z`) and the post-fix `finished_utc` field of `08-loadtest-cpu1.json` (`2026-06-22T08:51:13Z`) both parse as strict ISO-8601 UTC, are monotonic (start < finish), and span exactly 95.0 s, well within the 30-minute Strong window AND the 60-minute Fallback window. Unlike the `appinsights` lab which used dedicated traffic-completed timestamp files, this lab reads the temporal anchors directly from the load-test JSONs because the load-test runner records both `started_utc` and `finished_utc` natively; (c) no unexpected non-junk extras — Strong path requires exact match of the 15 canonical + 4 Phase B gate JSONs + `README.md`; Fallback path tolerates editor/OS junk (`.swp`, `.bak`, `.tmp`, `.swo`, `.orig`, `.DS_Store`, `Thumbs.db`) but still requires zero missing canonical files; (d) `evidence/README.md` cross-references all 4 Phase B gate JSON filenames literally (Strong) or just exists (Fallback). This sub-gate is the reason this README must literally name `14-cohort-integrity-gate.json`, `15-baseline-cpu-pressure-gate.json`, `16-recovery-materialization-gate.json`, and `17-single-variable-falsification-gate.json` — the verifier reads this file as a string and searches for each filename token to satisfy the Strong path.
- **Gate 15 — `cpu_throttling_h1_baseline_cpu_pressure_at_cpu_0.25_memory_0.5gi_single_replica_the_deterministic_cpu-bound_endpoint_produced_observable_cpu_pressure`** (4 sub-gates). Confirms (a) `01-app-config-before.json` records `cpu=0.25` AND `memory=0.5Gi` AND `minReplicas=1` AND `maxReplicas=1` AND `activeRevisionsMode=Single` (single-path, BOTH-not-OR — all five must hold simultaneously; the baseline envelope is the exact platform configuration under which the H1 measurement was taken); (b) `03-loadtest-cpu025.json.latency_ms.p95 > 100 ms` (the trivial-load floor — without strictly above 100 ms p95 the workload was too light to reproduce throttling and the run is invalid); (c) `03-loadtest-cpu025.json.requests_ok >= 95 AND requests_err == 0` (single-path, BOTH-not-OR — the measurement is not corrupted by transport failures); (d) `03-loadtest-cpu025.json.wall_clock_seconds > 5` (the load test ran long enough to actually exercise CPU rather than completing trivially). The gate is the necessary pre-condition for Gate 16 — without H1 passing, the recovery ratio in Gate 16 is meaningless because there was no pressure to recover from.
- **Gate 16 — `cpu_throttling_h2_recovery_materialization_raising_the_per-replica_resource_envelope_from_cpu_0.25_memory_0.5gi_to_cpu_1.0_memory_2.0gi_materialized_recovery`** (4 sub-gates). Confirms (a) `06-app-config-after.json` records `cpu=1.0` AND `memory in {"2Gi", "2.0Gi"}` AND `minReplicas=1` AND `maxReplicas=1` AND `latestRevisionName == 05-update-result.json.latestRevisionName` (cross-check pins the Phase 8 config-after read to the same revision that the Phase 6 update response named; memory normalization is platform-driven — the Bicep template requests `2.0Gi`, the `Microsoft.App` API normalizes it to `2Gi` on read); (b) `08-loadtest-cpu1.json.latency_ms.p95 < 0.5 * 03-loadtest-cpu025.json.latency_ms.p95` — strict less-than, equal-to-50% is borderline noise rather than recovery; the observed ratio is `0.3003` (`773.2 / 2574.8`), a 70.0% reduction, well below the 50% strict threshold; (c) `08-loadtest-cpu1.json.requests_ok >= 95 AND requests_err == 0` (single-path, BOTH-not-OR — the post-fix measurement is not corrupted by transport failures); (d) `07-revisions-after.json` (record-scoped iteration): exactly 2 revision records AND exactly 1 record has `trafficWeight==100` AND that record's `runningState ∈ {"Running", "RunningAtMaxScale"}` AND that record's name equals `06-app-config-after.json.latestRevisionName`. The `runningState` predicate explicitly excludes `"Deprovisioning"` so the cohort cannot count a tearing-down old revision as the active traffic target — the cohort's old revision `--8pz2nir` is in exactly this state (`active=true`, `runningState=Deprovisioning`, `trafficWeight=0`), so `active==true` alone would be unsafe.
- **Gate 17 — `cpu_throttling_h3_single_variable_falsification_the_cohort_cannot_attribute_recovery_to_anything_other_than_the_cpu_memory_delta`** (3 sub-gates). Confirms (a) on the shared-keys surface between `01-app-config-before.json` and `06-app-config-after.json` (shared_keys = `{cpu, memory, minReplicas, maxReplicas}`; `activeRevisionsMode` appears only in `01`, `latestRevisionName` appears only in `06` — a documented schema asymmetry of `az containerapp show` query projections), the set of keys whose values differ must equal exactly `{cpu, memory}`; the observed diff is exactly `{cpu: 0.25→1.0, memory: 0.5Gi→2Gi}` with `minReplicas=1` and `maxReplicas=1` unchanged; (b) revision lineage (record-scoped over `02-revisions-before.json` and `07-revisions-after.json`): the before-revision name `ca-cputhrottle-65svxr--8pz2nir` appears in `07` (any state, any traffic), `06.latestRevisionName == ca-cputhrottle-65svxr--0000001` appears in `07` with `trafficWeight==100`, AND that new name is NOT in `02` (the platform created a new revision during the update, did not edit the old one in place); (c) every revision name in both `02-revisions-before.json[*].name` and `07-revisions-after.json[*].name` starts with `13-deployment-outputs.json.containerAppName + "--"` (`ca-cputhrottle-65svxr--`), proving the cohort did not silently switch to a different Container App resource between before and after. The gate JSON carries an explicit `cohort_binding_note` field stating that the single-variable claim is bounded to the `(cpu, memory)` delta on the shared-fields surface of `01` and `06` and explicitly DOES NOT prove: (1) Kubernetes pod reuse — sub-gate (b) explicitly shows a NEW revision was created and the old revision is in `runningState==Deprovisioning`; (2) container image byte-identity — this is inferred from the fact that the Phase A `fix-and-capture.sh` issued `az containerapp update --cpu --memory` with no `--image`, `--command`, or `--args` flag, but the cohort does NOT capture image digests so this inference is a documented gap in the claim ceiling. This explicit binding enforces the Lab 21 Q1 "Bound every absence predicate to the failed-deployment cohort" directive in the resource-envelope falsification context.

## Honest disclosure — empirical platform behavior captured during this run

- **`04-metrics-cpu025.json` carries 60 `PT1M` timestamps with NO populated `Average` or `Maximum` samples.** Phase 5 of `trigger.sh` queries `az monitor metrics list --metric UsageNanoCores --interval PT1M --aggregation Average,Maximum` immediately after the 8.65-second baseline load test completes. Azure Monitor typically materializes per-minute aggregated platform metrics 1-3 minutes after the event, and `PT1M` aggregation further averages a short-duration load across a full minute. The captured JSON is the truthful Azure Monitor response at that moment: the metric definition exists, the timestamps for the requested window are present, but the `average` and `maximum` value fields are absent because the underlying samples had not yet materialized when the query ran. This is documented Azure Monitor behavior, not a script bug. The load-test latency in `03-loadtest-cpu025.json` is the lab's primary baseline evidence; the metric snapshot is kept as an honest record of the timing constraint.
- **`09-metrics-cpu1.json` has samples only on the trailing `2026-06-22T08:50:00Z` minute** (`Average=820660.5 nC`, `Maximum=852758.0 nC` — well below the 1,000,000,000 nC = 1.0 vCPU steady-state ceiling because the 2.85-second post-fix load test was averaged across the full 60-second minute, and the samples landed in the minute that started right after the resource-update activation). Gate 16 and Gate 17 anchor on the load-test latency, NOT on the metric values, so this short-event averaging behavior does not weaken the falsification. For a production diagnosis, wait 3-5 minutes after the load event before querying `UsageNanoCores`, and prefer the `Maximum` aggregation over `Average` for short-duration events.
- **Schema asymmetry between `01` and `06` is intentional** and documented as out-of-scope for Gate 17 sub-gate (a). The Phase A `trigger.sh` Phase 1 captures `01-app-config-before.json` via `az containerapp show --query "{activeRevisionsMode: properties.configuration.activeRevisionsMode, cpu: properties.template.containers[0].resources.cpu, memory: properties.template.containers[0].resources.memory, minReplicas: properties.template.scale.minReplicas, maxReplicas: properties.template.scale.maxReplicas}"`. The Phase 8 capture of `06-app-config-after.json` adds `latestRevisionName: properties.latestRevisionName` so the cohort can cross-check the post-update revision against the Phase 6 update response (`05-update-result.json.latestRevisionName`). The result is that `01` carries `activeRevisionsMode` (which is unchanged across the fix) but no `latestRevisionName` (which did not yet exist before the update), and `06` carries `latestRevisionName` (which is the new revision) but no `activeRevisionsMode` (because the schema is asymmetric by Phase A design). Gate 17 sub-gate (a) explicitly scopes the diff predicate to the shared keys `{cpu, memory, minReplicas, maxReplicas}` to handle this asymmetry cleanly.
- **Memory normalization is platform-driven.** The Bicep template requests `memory: '2.0Gi'`, and the `Microsoft.App` ARM API normalizes the value to `2Gi` on read. The captured `06-app-config-after.json` therefore records `"memory": "2Gi"`, while the `05-update-result.json` from the Phase 6 `az containerapp update --memory 2.0Gi` command does not echo the memory field back. Gate 16 sub-gate (a) accepts both variants `{"2Gi", "2.0Gi"}` for the memory predicate to immunize against this normalization. The earlier `memory: "0.5Gi"` value is not normalized (since `0.5Gi` is not equivalent to any integer-Gi value) and appears as `"0.5Gi"` literally in `01`.
- **The old revision `--8pz2nir` is in `runningState=Deprovisioning` with `active=true` and `trafficWeight=0` in the post-fix `07-revisions-after.json`.** This is documented `Microsoft.App` behavior — under `activeRevisionsMode: Single`, when a new revision becomes active, the platform marks the new revision with `trafficWeight=100` and starts the asynchronous tear-down of the old revision while keeping `active=true` on the old revision until tear-down completes. Gate 16 sub-gate (d) uses the predicate `trafficWeight==100 AND runningState ∈ {"Running", "RunningAtMaxScale"}` (NOT `active==true`) precisely to exclude the tearing-down old revision from being counted as the traffic-100 holder.
- **Image byte-identity is NOT cohort-evidenced.** The single-variable claim in Gate 17 is bounded to the `(cpu, memory)` delta on the shared-keys surface of `01` and `06`. The cohort does not capture image digests on either revision, so the claim that the application image bytes are identical across the revisions is inferred from the Phase A fix command (`az containerapp update --cpu 1.0 --memory 2.0Gi` with no `--image`, `--command`, or `--args` flag) but not directly evidenced. This is documented as a gap in the claim ceiling inside the gate JSON's `cohort_binding_note` field. A future evidence-pack revision could close this gap by adding a `revisions[*].properties.template.containers[0].image` capture, but this lab's canonical cohort does not.
- **Pod reuse is NOT claimed.** Under `activeRevisionsMode: Single`, `az containerapp update --cpu --memory` creates a new revision (the Phase 6 update response in `05-update-result.json` returns `latestRevisionName=ca-cputhrottle-65svxr--0000001` and `provisioningState=Succeeded`), and the new revision activates a fresh replica. The single-variable claim is bounded to the `(cpu, memory)` delta only — Gate 17 sub-gate (b) explicitly shows the new revision did not exist in `02-revisions-before.json` and is the `trafficWeight==100` holder in `07-revisions-after.json`. Any operator reasoning about whether the Kubernetes pod is reused or recreated is out-of-scope for this lab.
- **Activation-window timing is NOT a generalized claim.** The cohort captured one 15-second revision-state transition (`08:50:45Z runningState=Activating` → `08:51:00Z runningState=RunningAtMaxScale`) during this single Phase A reproduction. This is one observation, not a multi-run timing bound. Azure Container Apps revision-activation latency can vary with image size, replica count, and regional load; a longer activation-window lab would be needed to bound the platform's typical activation behavior. Gate 17's `cohort_binding_note` explicitly drops the claim that the platform "generally activates the new revision within 15 seconds" for this reason.
- **`validation.az_cli.last_tested: 2026-06-22` is preserved across the Phase B overlay** per the `AGENTS.md` "Never fabricate validation dates" rule. The Phase B verifier ran on 2026-06-26 against the 2026-06-22 cohort; it did not re-execute the live-Azure capture, so the original validation date is the only date that reflects when the cohort was empirically observed.
- **The workload image is `python:3.12-slim` from public Docker Hub, NOT from a private ACR.** No ACR is provisioned by this lab's `infra/main.bicep` because the CPU-burn HTTP server is small enough to inline as a Python `-c` argument in the Bicep `command` field. This is intentional — for a pure CPU-throttling demonstration, the workload's container-pull path is irrelevant, so cutting ACR out of the lab reduces both cost and capture surface. A separate lab series (`acr-network-path-*`) covers the ACR-pull path.
- **No Application Insights is provisioned.** This lab's evidence comes entirely from the load-test HTTP client (latency percentiles) and the platform `UsageNanoCores` metric (subject to the timing limitation documented above). The Container Apps environment is deployed without `appInsightsConfiguration`, the application image has no OpenTelemetry SDK, and no `AppRequests` or `AppTraces` rows materialize in any workspace. The lab's falsification target is the per-replica `(cpu, memory)` envelope, NOT telemetry export — that is a separate lab (`appinsights-connection-string-missing`).
