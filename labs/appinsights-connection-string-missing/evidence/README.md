# Evidence pack — `appinsights-connection-string-missing` lab

This directory carries the committed canonical evidence cohort for the `appinsights-connection-string-missing` lab, anchored on the live-Azure deployment in `koreacentral` on 2026-06-22. The cohort consists of 20 canonical Phase A files (2 script logs + 6 before-fix captures + 5 after-fix captures + 4 KQL captures + 3 image-lineage captures), plus 4 Phase B falsification-gate JSONs emitted by `labs/appinsights-connection-string-missing/verify.sh`, plus this README. All Azure identifier PII has been replaced with documentation-safe placeholders (subscription/tenant/object GUIDs scrubbed to the zero-GUID `00000000-0000-0000-0000-000000000000`, employee aliases scrubbed to `demouser`, employee emails scrubbed to `user@example.com`, FQDN tenant-domain segments scrubbed to `envdomain-aaaaaaaa`, and the App Insights connection string value redacted at capture time by the scripts themselves so the secret is never committed).

## Capture timeline

Three carry-over framings to make the evidence pack readable:

1. **The canonical image is `:v3`, not `:v1` or `:v2`.** The `:v1` image calls `configure_azure_monitor()` UNGUARDED and crashes the gunicorn worker on import when `APPLICATIONINSIGHTS_CONNECTION_STRING` is unset (`ValueError: Instrumentation key cannot be none or empty.`); the `:v2` image fixed the crash with an env-var presence guard but imported Flask with `from flask import Flask` before SDK init, which silently broke Flask auto-instrumentation. The `:v3` image is the canonical scenario: env-var guard plus `import flask` as a module plus explicit `logger_name`. The A1/A2/A3 captures are kept under this evidence directory as image-lineage documentation, NOT as part of the `:v3` before/after falsification cohort. See the lab `README.md` "Why the canonical image is `:v3`" section for the full carry-over.
2. **This lab uses the SDK instrumentation path, not the environment-level OpenTelemetry agent.** The Container Apps environment is intentionally deployed without `appInsightsConfiguration` on the environment resource, so `06-env-telemetry-config.json` captures the expected `ERROR: The containerapp environment 'cae-appiconn-x2lcxd' does not have app insights enabled.` This is a documented design choice — the lab's falsification target is the per-container env var `APPLICATIONINSIGHTS_CONNECTION_STRING` consumed by `azure-monitor-opentelemetry==1.6.4` inside the Python app, not the env-level agent that auto-instruments without code changes.
3. **Recovery is in-place: the single delta is the env var.** The before-fix and after-fix revisions (`--0000006` and `--0000007`) run the byte-identical `:v3` image on the same registry (`acrappiconnx2lcxd.azurecr.io`) with the same container name (`app`), the same resources (`cpu=0.5`, `memory=1Gi`, `ephemeralStorage=2Gi`), and the same image type (`ContainerImage`). The only difference is the presence of `APPLICATIONINSIGHTS_CONNECTION_STRING` in `properties.template.containers[0].env`. This single-variable property is what Gate 19 anchors on.

Capture phases:

- **Phase 1 — Provisioning** (`infra/main.bicep`): one resource group, one Log Analytics workspace (PerGB), one workspace-based Application Insights, one ACR Basic registry, one Container Apps Environment (Consumption, no env-level telemetry agent), and one Container App seeded with the placeholder image `mcr.microsoft.com/azuredocs/containerapps-helloworld:latest` (no SDK at all). System-assigned identity plus `AcrPull` role assignment on the lab ACR. The instrumented `:v3` image is built directly inside the lab's ACR with `az acr build` so no local Docker daemon is required.
- **Phase 2 — Failure reproduction** (`trigger.sh`, 93 lines): switch the Container App to the instrumented `:v3` image WITHOUT setting `APPLICATIONINSIGHTS_CONNECTION_STRING`, poll up to 3 minutes for the new revision to reach `provisioningState=Provisioned` with `replica=Running` and `container=Running`, capture the env state (expect `env=null`) to `01-env-before-fix.json`, resolve the Container App FQDN, generate 20 sequential `curl` requests (all HTTP 200), record the traffic-completed UTC timestamp to `02-traffic-completed-before-fix.txt`, sleep 240 seconds for App Insights ingestion latency, query `AppRequests` and `AppTraces` aggregates (both expected empty) to `03-` and `04-`, capture the full Container App config to `05-`, and capture the env-level telemetry config (expected ERROR — intentional) to `06-`.
- **Phase 3 — Recovery and re-capture** (`fix-and-capture.sh`, 224 lines): read the App Insights connection string from the deployed resource (value never echoed to logs), apply the fix with `az containerapp update --set-env-vars APPLICATIONINSIGHTS_CONNECTION_STRING=...`, poll up to 3 minutes for the new revision to reach `provisioningState=Provisioned` with `replica=Running` and `container=Running`, capture the env state (expect the env var name present, value redacted) to `07-env-after-fix.json`, resolve the FQDN, generate 20 fresh `curl` requests (all HTTP 200), record the traffic-completed UTC timestamp to `08-traffic-completed-after-fix.txt`, sleep 240 seconds for ingestion latency, query `AppRequests` and `AppTraces` aggregates (both expected populated) to `09-` and `10-`, capture the full Container App config to `11-`, capture per-5-minute `AppRequests` timeline to `12-`, capture per-message `AppTraces` (`take 25`) to `13-`, capture all 8 lab revisions with a `hasConnStr` decorator computed in a `python3 - <<'PY'` heredoc to `14-`, capture per-request `AppRequests` detail to `15-`, and print the recovery `PASS` marker.
- **Verify Phase — Phase B gate emission** (`verify.sh`, 1360 lines, 2026-06-25): a pure file processor that reads only the 20 canonical Phase A files under this directory and emits four derived gate JSONs (`16-cohort-integrity-gate.json`, `17-failure-attribution-gate.json`, `18-recovery-materialization-gate.json`, `19-single-variable-falsification-gate.json`). The verifier never calls `az`, never touches the live tenant, and is fully replayable from the committed cohort. Run via `bash labs/appinsights-connection-string-missing/verify.sh`.

## Why this is an SDK-instrumentation-presence falsification lab, not a KQL-attribution lab

The acr-pull-failure lab uses a deployment-level failure model (the container never starts, the `MANIFEST_UNKNOWN` token in the deployment operation record is the smoking gun, and `ContainerAppSystemLogs_CL` is never materialized in the workspace). This lab is the opposite: the container DOES start, every HTTP request DOES return 200, and the failure mode is the absence of telemetry rows in App Insights. The four Phase B gates encode four orthogonal claims that together prove the env-var-presence falsification:

- **Gate 16 — `appinsights_h_cohort_integrity_evidence_cohort_is_internally_consistent_and_temporally_coherent`** (4 sub-gates). Confirms (a) all 20 canonical Phase A files exist on disk — Strong path requires all 20, Fallback path requires `>= 18`; (b) temporal coherence — the before-fix traffic-completed timestamp (`02-traffic-completed-before-fix.txt:1`, `2026-06-22T05:02:57Z`) and the after-fix traffic-completed timestamp (`08-traffic-completed-after-fix.txt:1`, `2026-06-22T05:09:45Z`) both parse as strict ISO-8601 UTC (`YYYY-MM-DDTHH:MM:SSZ`), are monotonic, and span 408.0 s, well within the 30-minute Strong window AND the 60-minute Fallback window. Both files are single-line runtime captures from `trigger.sh` and `fix-and-capture.sh` respectively — there are no backfilled timestamps in this lab; (c) no unexpected non-junk extras — Strong path requires exact match of the 20 canonical + 4 Phase B gate JSONs + `README.md`; Fallback path tolerates editor/OS junk (`.swp`, `.bak`, `.tmp`, `.swo`, `.orig`, `.DS_Store`, `Thumbs.db`) but still requires zero missing canonical files (the Lab 19 P0 `unexpected_non_junk_extras` predicate + Lab 20 PR #279 fallback-integrity directive); (d) `evidence/README.md` cross-references all 4 Phase B gate JSON filenames literally (Strong) or just exists (Fallback). The A1/A2/A3 image-lineage captures are recognized as canonical cohort members; they are NOT extras.
- **Gate 17 — `appinsights_h1_failure_attribution_before_fix_cohort_is_fully_attributable_to_missing_applicationinsights_connection_string`** (4 sub-gates). Confirms (a) `01-env-before-fix.json` shows `env=null` AND the image is `acrappiconnx2lcxd.azurecr.io/hellotelemetry:v3` AND the container name is `app` (record-scoped on the single container); (b) `03-ai-requests-before-fix.json` `tables[0].rows == []` (no `AppRequests` rows under any `cloud_RoleName`); (c) `04-ai-traces-before-fix.json` `tables[0].rows == []` (no `AppTraces` rows under any `cloud_RoleName`); (d) `00-trigger-run.txt` records 20 lines matching `request NN → HTTP 200` AND a total line count of 93 (Strong path requires both, Fallback path requires `>= 18` HTTP 200 lines). The gate JSON carries an explicit `cohort_binding_note` field stating that **sub-gates (b) and (c) are absence-style predicates that are interpretable as missing-env-var evidence ONLY in conjunction with sub-gate (a) and sub-gate (d)** — without the bound cohort framing, an empty `AppRequests` table could be misread as "no traffic was generated" rather than "traffic was generated but the SDK was not wired". This explicit binding enforces the Lab 21 Q1 "Bound every absence predicate to the failed-deployment cohort" directive in the env-var-presence context.
- **Gate 18 — `appinsights_h2_recovery_materialization_adding_applicationinsights_connection_string_materialized_end_to_end_telemetry`** (5 sub-gates). Confirms (a) `07-env-after-fix.json` `envNames` contains exactly `["APPLICATIONINSIGHTS_CONNECTION_STRING"]` AND the image is still `acrappiconnx2lcxd.azurecr.io/hellotelemetry:v3` AND the container name is still `app` (single-path, BOTH-not-OR); (b) `09-ai-requests-after-fix.json` has exactly one row in `tables[0].rows` with `cloud_RoleName == "unknown_service"` AND `requestCount == 20` (single-path, record-scoped iteration, BOTH-not-OR per Lab 19 P0); (c) `10-ai-traces-after-fix.json` has exactly one row with `cloud_RoleName == "unknown_service"` AND `traceCount == 21` (single-path, BOTH-not-OR — the `21` decomposes into 1 startup `Azure Monitor configured: telemetry export enabled` trace plus 20 per-request `/ endpoint hit (conn_str_present=True)` traces); (d) `13-ai-traces-messages-after-fix.json` carries 21 trace records with the startup-log content at `message_column_index=1` AND 20 endpoint-hit records (record-scoped iteration, parses JSON instead of grepping per Lab 15 lesson 34); (e) `00-verify-run.txt` records 20 lines matching `request NN → HTTP 200` AND a total line count of 111 AND the literal recovery marker `PASS: After adding APPLICATIONINSIGHTS_CONNECTION_STRING` is present. The gate is single-path on each sub-gate — there are no dual-path Strong/Fallback alternatives on Gate 18 because each sub-gate is checking the structured shape of a canonical capture file, not a tolerance-bounded observation.
- **Gate 19 — `appinsights_h3_single_variable_falsification_recovery_is_in_place_single_env_var_delta_same_image_container_registry`** (3 sub-gates). Confirms (a) the before-fix and after-fix `properties.template.containers[0]` records are byte-identical after stripping the `env` field — same image (`acrappiconnx2lcxd.azurecr.io/hellotelemetry:v3`), same container name (`app`), same `resources` block, same `imageType` (`ContainerImage`); (b) `14-revisions-lifecycle.json` shows the before-fix revision (`--0000006`) with `active=false`, `hasConnStr=false`, `createdTime=2026-06-22T05:01:48+00:00`, image `:v3`; and the after-fix revision (`--0000007`) with `active=true`, `hasConnStr=true`, `createdTime=2026-06-22T05:08:34+00:00`, image `:v3` — record-scoped on the cohort pair, NOT on the revision-history limit (the lab carries 8 revisions in total, of which the first 6 are image-lineage and cohort-irrelevant); (c) the registry prefix (`image.split('/')[0]`) is byte-identical across both revisions (`acrappiconnx2lcxd.azurecr.io`). **Tear-down / no-recreation claims are dropped per the Lab 21 Q1 directive** as not soundly evidenced by this cohort — the cohort cannot prove the underlying resource was NOT torn down and recreated with the same name in the same span, so the gate does not assert that. The single-variable claim is anchored to the env-var delta on the same image on the same registry on the same container name, NOT to a stronger no-recreation claim.

## Honest disclosure — empirical platform behavior captured during this run

- **`A1`, `A2`, and `A3` document the `:v1` UNGUARDED SDK crash, NOT the canonical `:v3` cohort.** They are kept here as image-lineage documentation so the lab itself records what was tried and why the canonical image is `:v3`. The `:v1` image crashes the gunicorn worker on import with `ValueError: Instrumentation key cannot be none or empty.` raised inside `configure_azure_monitor()` at `/usr/local/lib/python3.11/site-packages/azure/monitor/opentelemetry/exporter/_connection_string_parser.py:106` (gunicorn worker pid 8 exits with code 3 "Worker failed to boot", `runningState=Waiting`, `runningStateDetails="Container is waiting with reason: CrashLoopBackOff on legion."`, `restartCount=7`). This is a DIFFERENT failure mode (availability loss, not silent observability gap) and is intentionally NOT part of the canonical `:v3` before/after falsification.
- **`06-env-telemetry-config.json` carries an expected `ERROR`, not a script bug.** The Container Apps environment is intentionally deployed without `appInsightsConfiguration` because this lab targets the SDK code-path inside the application image, not the platform-managed OpenTelemetry agent. The `ERROR: The containerapp environment 'cae-appiconn-x2lcxd' does not have app insights enabled.` line is captured to document that the platform-level agent is OFF, so the SDK path is the only telemetry path under test.
- **The canonical image is `:v3`, not `:v1` or `:v2`.** The lab `README.md` "Why the canonical image is `:v3`" section carries the full lineage. The `:v2` source file is intentionally not committed to the repo (it documented a Flask import-ordering subtlety that surfaced during lab development); no per-revision `AppRequests`/`AppTraces` capture file exists for `:v2`. The canonical falsification pair is the `:v3` before/after run (revisions `--0000006` and `--0000007`), not a `:v2`/`:v3` comparison.
- **`validation.az_cli.last_tested: 2026-06-22` is preserved across the Phase B overlay** per the `AGENTS.md` "Never fabricate validation dates" rule. The Phase B verifier ran on 2026-06-25 against the 2026-06-22 cohort; it did not re-execute the live-Azure capture, so the original validation date is the only date that reflects when the cohort was empirically observed.
- **The 240-second App Insights ingestion-latency sleep is hardcoded in both `trigger.sh` and `fix-and-capture.sh`.** App Insights typically ingests fresh telemetry within 2 to 5 minutes; both scripts sleep 240 s after sending traffic before running the KQL query. If a re-run of the live scripts shows 0 rows in `AppRequests` after the sleep, the lab `README.md` documents a 2-minute hand-retry recipe. The Phase B verifier does NOT re-introduce this latency — it reads the already-captured KQL files.
- **`cloud_RoleName == "unknown_service"` is the empirically observed `cloud_RoleName` for this lab's Python image.** The `azure-monitor-opentelemetry==1.6.4` distro does not infer a meaningful service name from the Container App or revision name; it defaults to `unknown_service` unless `OTEL_SERVICE_NAME` is set in the container env. This lab intentionally does not set `OTEL_SERVICE_NAME` because the falsification target is the connection string env var alone — adding a second env var would dilute the single-variable claim. Gate 17 and Gate 18 both anchor on the literal string `"unknown_service"` because that is what the captured JSON contains; a real production app would set `OTEL_SERVICE_NAME` and would see a different role name.
- **`14-revisions-lifecycle.json` carries 8 revisions, of which only the last 2 (`--0000006` and `--0000007`) are the canonical Phase B cohort pair.** Revisions `--vfjk66n`, `--0000001`, `--0000002`, `--0000003`, `--0000004`, and `--0000005` are image-lineage artifacts (placeholder helloworld, `:v1` crash documentation, `:v2` development iterations) and are NOT consumed by any Phase B gate predicate. Gate 19 sub-gate (b) explicitly scopes its record-scoped iteration to the cohort pair only — the presence of the 6 older revisions does NOT pollute the single-variable claim.
- **Recovery is anchored to a single env var delta plus byte-identical container config.** Gate 19 sub-gate (a) confirms the `template.containers[0]` records are byte-identical after stripping `env`. Sub-gate (c) confirms the registry prefix is identical. Sub-gate (b) confirms the env-var presence flipped from `hasConnStr=false` to `hasConnStr=true`. The gate does not claim the container was not torn down and recreated; the underlying revision activation mechanism is a black box that the cohort cannot pierce.

## Reproducibility

The Phase A live-Azure capture is reproduced end-to-end by running the lab's three scripts in order (`trigger.sh`, `fix-and-capture.sh`, `cleanup.sh`) against a fresh deployment of `infra/main.bicep`. See the lab `README.md` "Quick Start" section for the full bash recipe. The Phase B falsification gates are reproduced offline from the committed cohort:

```bash
cd labs/appinsights-connection-string-missing/
bash verify.sh
# Reads the 20 canonical Phase A files + this README, emits the 4 gate JSONs (16/17/18/19),
# prints a per-gate verdict table, and exits 0 on all-PASS or 2 on any-FAIL.
# The verifier is a pure file processor — it never calls `az`, never touches the live tenant,
# and is fully replayable.
```
