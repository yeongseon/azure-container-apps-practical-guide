# Lab: replica-node-spread

Empirical test of how Container Apps distributes replicas across underlying shared-kernel contexts on **Consumption** vs **Dedicated D8** workload profiles within a single Container Apps environment.

The lab uses observable proxy signals (`/proc/sys/kernel/random/boot_id`, `/proc/uptime`, microcode) to infer whether replicas share a kernel context (consistent with the same node) or do not (consistent with different nodes). The lab does **NOT** claim direct node identification — see the [lab guide](../../docs/troubleshooting/lab-guides/replica-node-spread.md) for the full epistemic discipline.

This experiment was Oracle-design-reviewed (session `ses_14b7919caffe2lB37qNy19bGl5`) with 10 mandated modifications, all incorporated below.

## Structure

```text
labs/replica-node-spread/
├── infra/
│   ├── main.bicep                 # env (Consumption + D8) + 2 apps + ACR-pull UAMI
│   └── main.parameters.json
├── app/
│   ├── Dockerfile                 # cbl-mariner + bash + jq
│   └── diag.sh                    # emits one JSON line per call (boot_id, uptime, microcode, ...)
├── deploy.sh                      # RG + ACR + az acr build + Bicep deploy
├── verify.sh                      # post-deploy health checks + diag.sh probe
├── trigger.sh                     # full scale ladder + top-scale × 3 repeat
├── sample.sh                      # enumerate replicas, run diag.sh on each, JSONL output
├── h3-falsification.sh            # Oracle-mandated proxy validation (gate for H1/H2)
├── analyze.py                     # JSONL → markdown summary (boot_id histogram + clustering)
├── cleanup.sh                     # destructive teardown
├── evidence/                      # raw JSONL + analysis outputs (permanent, do not delete)
│   └── .gitkeep
└── README.md
```

## Prerequisites

- Azure subscription with Container Apps + ACR Basic available in target region
- Region supporting **workload profiles** and **D8 SKU** (recommended: `koreacentral`, `eastus`)
- Azure CLI `2.60+` with `containerapp` extension (`az extension add --name containerapp`)
- Sufficient Consumption-profile quota: 30 × 0.25 vCPU = 7.5 vCPU peak
- Sufficient Dedicated D-series quota for 1 × D8 node

## Quick start

```bash
export RG="rg-aca-rns-lab"
export LOCATION="koreacentral"
export ACR_NAME="acrrnslab$(date +%s | tail -c 6)"

./deploy.sh
./verify.sh

./h3-falsification.sh
python3 analyze.py evidence/h3-same-replica-*.jsonl --mode h3a --output evidence/h3a-summary.md
python3 analyze.py evidence/h3-restart-*.jsonl --mode h3b --output evidence/h3b-summary.md

./trigger.sh
python3 analyze.py evidence/consumption-*.jsonl --output evidence/consumption-summary.md
python3 analyze.py evidence/dedicated-*.jsonl --output evidence/dedicated-summary.md

./cleanup.sh
```

## Evidence retention policy

**The `evidence/` directory is permanent and committed to the repository.** User mandate:

## Cost containment

| Resource | Approx. cost during experiment |
|---|---|
| 30 × Consumption replicas @ 0.25 vCPU × ~30 min | ~3.75 vCPU-hours |
| 1 × D8 node × ~1.5 hours (ramp + samples + idle) | ~$0.60 |
| ACR Basic, prorated | ~$0.01/hour |
| LAW ingestion (debug only) | ~$0.10 |

The Bicep template stamps `expires-at` 24 hours from deployment time. **Always run `./cleanup.sh` immediately after analysis.**
