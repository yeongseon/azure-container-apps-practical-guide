# Evidence Pack Provenance

This directory contains the raw evidence artifacts for the `replica-node-spread` lab. The lab is a **two-profile differential observation** — a single Container Apps environment hosts two apps (`app-consumption` on the Consumption workload profile, `app-dedicated-d8` on a Dedicated D8 profile, 8 vCPU / 32 GiB / 1 node) and the experiment sweeps each app across scale targets {1, 3, 10, top} (top = 30 for Consumption, 10 for Dedicated D8) while polling a `/diag` endpoint that returns kernel-context proxies (`boot_id`, `uptime_seconds`, `boot_time_estimate_ms`, `kernel_release`, `microcode`, `cpu_model`). The Consumption top scale (30) is repeated **3 times** to test concurrence; the Dedicated D8 top scale (10) is repeated **3 times** for the same reason. The cohort was captured in a single live-Azure window on **2026-06-14** against RG `rg-aca-rns-lab` (Korea Central, az-cli 2.83.0, containerapp extension current at capture time). All files are PII-scrubbed (subscription/tenant GUIDs replaced with the zero-GUID placeholder, employee aliases replaced with `demouser`, employee emails replaced with `user@example.com`, Log Analytics workspace IDs replaced with the zero-GUID placeholder, local user paths replaced with `/Users/demouser`).

## Capture timeline

The lab evidence was captured by the lab's own `trigger.sh` orchestrator (which calls `falsify.sh` for the H3 gate, then `scale.sh` + `sample.sh` for each `{profile, scale, run}` cell) plus `analyze.py` for the per-run summary. The Phase B refactor in this commit **reuses that 2026-06-14 evidence as the canonical baseline** rather than redeploying — three reasons drive that decision:

1. **Oracle Lab 19 directive (2026-06-24, bg_021159d5)** — "Pick `replica-node-spread` over `zone-redundancy-best-effort`. Admit only this cohort: `analysis-summary.{json,md}`, `h3-20260614-143432.{jsonl,verdict.txt}`, `consumption-scale-*.jsonl`, `dedicated-d8-scale-*.jsonl`. Treat `h3-20260614-143432` as the cohort anchor. If a raw file's provenance to that bundle is unclear, exclude it rather than infer continuity."
2. **Cost minimization (carry-over user constraint)** — A redeploy would re-provision one Container Apps Environment with two workload profiles (Consumption + Dedicated D8), one ACR, and two Container Apps each scaled to 30 / 10 replicas for ~45 minutes, costing ~$2-3 USD per re-run, when the existing evidence already contains every field the new four-gate predicate suite needs.
3. **Scheduler-placement determinism is not guaranteed across re-runs** — The Consumption top-scale-30 run captured 27 boot_time clusters across 30 replicas (i.e. 3 replicas were co-located on existing kernel contexts) in **all three** repeats — concurrent — but a fresh re-run might land 25 or 29 clusters depending on the scheduler's free-capacity decisions at that moment. The Phase B gate thresholds are written to be re-run-tolerant (concurrence is checked across **repeats within a run**, not across runs in different capture windows) but the committed evidence is what the gates evaluate.

## Cross-profile differential proof — why this is not a deterministic-spread lab

This lab does NOT claim "Container Apps deterministically spreads replicas one per node on Consumption." That claim is too strong for what `/diag` can prove: the platform exposes kernel-context proxies (boot_id / uptime / boot_time_estimate_ms), not the underlying `Microsoft.Compute` node id. Two replicas with **different** boot_id values are necessarily on different kernel contexts (and almost certainly on different nodes); two replicas with the **same** boot_id are on the same kernel context (and very likely on the same node). The lab's headline claim is intentionally narrow: **observed node-spread behavior under this test matrix**, capped at `[Strongly Suggested]`. The four gates encode that contrast:

- **Gate 10 (cohort integrity, 4 sub-gates) — the cohort is uncontaminated.** Anchor file + verdict exist (sub-gate a), 100% of 1117 records parse with the 4 required keys `{boot_id, replica_name, uptime_seconds, run_id}` (sub-gate b — the minimal common set across anchor and scale records, because anchor records carry `phase` while scale records carry `profile`/`scale_target`/`sample_index`), 100% of records carry the date prefix `20260614` (sub-gate c — proves single-window provenance), and the evidence directory has exactly the 15 canonical files plus the 4 Phase B-emitted gate JSONs and this README (sub-gate d — no foreign artifacts from other labs or stray test runs).

- **Gate 11 (matrix coherence, 4 sub-gates) — the test matrix is one-to-one.** Each of the 11 scale files decomposes uniquely into a `{profile, scale, run}` cell (sub-gate a — the explicit `PROFILE_FILENAME_TO_RECORD` case map handles the filename `consumption` ↔ record `"Consumption"` mismatch), zero duplicate cells across the 11 files (sub-gate b), all 11 files reconcile bit-exact with the `analysis-summary.json` entry for the same filename (sub-gate c — the recompute path reads raw JSONL and rebuilds unique-replica / unique-boot-id / boot_time-cluster counts independently, then compares to the summary's `boot_time_clusters` field), and the H3 verdict text recomputes from the anchor JSONL with all 4 falsifiable checks reproducing (sub-gate d — recomputed 1 unique boot_id, monotonic uptime sequence `[2054.0, 2064.13, 2074.24, 2084.36, 2094.48]`, 7 ms boot_time_estimate_ms span).

- **Gate 12 (claim eligibility, 5 sub-gates) — claims stay within evidence ceiling.** Headline claim level is `Strongly Suggested` (sub-gate a — the Strong-path check refuses to allow any claim string containing the word `Observed` for placement, and the Fallback path checks the gate JSON's `claim_level` field directly), Consumption and Dedicated D8 are tabulated as **separate** profiles with separate per-scale data (sub-gate b — no cross-profile pooling, per Oracle directive), the 3 Consumption-n30 and 3 Dedicated-D8-n10 repeats show variability across runs (sub-gate c — Consumption-30 has 3 unique cluster-center vectors across runs, Dedicated-D8-10 has 2 unique cluster-center values across runs, proving the captures are genuinely independent rather than identical replays), **counterexamples are surfaced** rather than suppressed (sub-gate d — 7 known co-location cases logged: 3 Consumption-n30 runs with 27 clusters / 30 replicas = 3 co-located, 1 Dedicated-n3 with 1 cluster / 3 replicas = 2 co-located, 3 Dedicated-n10 runs with 1 cluster / 10 replicas = 9 co-located each), and per-file recompute of cluster counts from raw JSONL matches the summary in 11/11 files with zero discrepancies (sub-gate e — falsifies a "summary-was-fudged" theory).

- **Gate 13 (packaging, 3 sub-gates) — the pack is self-contained and re-verifiable.** All three prior gate JSONs exist before this gate runs (sub-gate a — `verify.sh` orchestration order is correct), this README references all four gate filenames `10-cohort-integrity-gate.json` / `11-matrix-coherence-gate.json` / `12-claim-eligibility-gate.json` / `13-packaging-gate.json` (sub-gate b — a reviewer reading this README can locate every gate output), and the 15 canonical files are present with no missing entries (sub-gate c).

The Consumption-vs-Dedicated **differential** is what makes the lab falsifiable: Consumption top-scale-30 produced **27 distinct boot_time clusters** (Strong-suggested ~27 nodes, with 3 co-locations), while Dedicated-D8 top-scale-10 produced **1 boot_time cluster across all 10 replicas in all 3 repeats** (Strong-suggested single-node placement, as expected for a single-node D8 profile). If a future re-run produced the inverse — Dedicated-D8 multi-cluster or Consumption single-cluster — the gates would still pass (the gates don't hardcode the ordering), but the operator narrative would need to be revised. The 7 surfaced counterexamples (Consumption-30 partial co-location, Dedicated-3 full co-location) prove the lab does NOT claim "always one replica per node" — it claims "the Consumption profile's scheduler distributes replicas across **many** kernel contexts at scale 30, while the Dedicated D8 profile's scheduler co-locates all replicas on **one** kernel context at scale 10, under this test matrix."

## Honest disclosure — empirical platform behavior captured during this run

The following observations surfaced during the 2026-06-14 historical run and are documented here so a future operator (or Oracle reviewer) can interpret the raw evidence without re-discovering each gap:

- **The H3 anchor proves intra-replica stability, NOT cross-replica distinctness.** `h3-20260614-143432.jsonl` has 5 samples all hitting the same replica (`app-consumption--0000001-65f97bcd78-k4j8r`) with the same `boot_id`. This proves the kernel-context proxy is **stable for a given replica** (necessary precondition for using boot_id as a node-identity inference), but it does NOT prove two different replicas with different boot_ids are on different nodes. The cross-replica claim is what Gates 11/12 evaluate against the scale sweeps, NOT against the anchor.

- **The test matrix is asymmetric by design.** Consumption was swept through `{1, 3, 10, 30}` while Dedicated D8 was swept through `{1, 3, 10}` only. The asymmetry is driven by Azure subscription quota: a single Dedicated D8 node provides 8 vCPU, and at 0.25 vCPU per replica × 10 replicas = 2.5 vCPU per replica, the lab can fit 10 D8 replicas on one node but not 30. Consumption has no such per-node ceiling because it provisions additional kernel contexts on demand. The gates DO NOT compare Consumption-30 vs Dedicated-30 directly (that comparison is impossible); they compare each profile's **top-scale concurrence behavior** separately.

- **CPU model divergence is empirical, not designed.** Consumption replicas reported `Intel(R) Xeon(R) Platinum 8370C CPU @ 2.80GHz` in `/proc/cpuinfo`; Dedicated D8 replicas reported `AMD EPYC 7763`. The lab does not select hardware vendor; this divergence is what Azure happened to allocate on 2026-06-14 in Korea Central. A future re-run might land Consumption on AMD and Dedicated D8 on Intel; the gates do NOT key off CPU vendor — they key off `boot_id` and `boot_time_estimate_ms`, which are kernel-instance proxies independent of underlying hardware.

- **Evidence ceiling is `[Strongly Suggested]`, not `[Observed]`.** `/diag` exposes kernel-context proxies (boot_id, uptime, boot_time_estimate_ms), NOT the underlying `Microsoft.Compute` node id. The Container Apps control plane does not surface node identity to the runtime. A single-cluster verdict at Dedicated D8 top scale is **consistent with** single-node placement; it is not proof that ACA cannot place D8 replicas on multiple nodes in some other configuration. A multi-cluster verdict at Consumption top scale is **consistent with** multi-node placement; the scheduler may still co-locate replicas on the same node when load patterns permit (which is exactly what the 27/30 counterexample observed — see next point). Gate 12 sub-gate (a) Strong path enforces this ceiling by refusing any headline claim containing the word `Observed` for placement.

- **The 5000 ms boot_time-cluster gap threshold is empirically chosen, not platform-documented.** `analyze.py` clusters replicas by `boot_time_estimate_ms = sample_timestamp - uptime_seconds`. Two replicas land in the same cluster if their estimates differ by < 5000 ms. This threshold absorbs the natural jitter in sampling timestamps (each `/diag` call adds 1-2 ms of HTTP round-trip variance, and `uptime` is in seconds with ~10 ms precision on Linux). A 1000 ms threshold would over-split (treating sampling jitter as different nodes); a 60000 ms threshold would under-split (collapsing genuinely-different nodes into one cluster). The threshold is documented at the top of `analyze.py` and recorded in every gate JSON's `thresholds.boot_time_cluster_gap_ms` field.

- **Counterexample #1 — Consumption scale-30 has 3 co-located replicas per run, in all 3 repeats.** Every Consumption-30 run captured 30 unique `boot_id` values (so 30 distinct kernel instances) but only 27 distinct boot_time clusters. That means 3 pairs of replicas were started so close together in time (within 5000 ms) that their `boot_time_estimate_ms` estimates collapsed into the same cluster — Strong-suggested same-node placement for those pairs. The cluster-center values are stable across all 3 repeats (within ~10 ms), suggesting these are persistent kernel instances that the scheduler revisited. This is documented in Gate 12 sub-gate (d) as `co_located_replica_count: 3` per run.

- **Counterexample #2 — Dedicated D8 scale-3 has all 3 replicas on 1 boot_id (full co-location).** When `app-dedicated-d8` was scaled to 3 replicas, the platform produced 3 unique `replica_name` values but exactly **1** unique `boot_id` — all 3 replicas were placed in the same kernel context on the single D8 node. Same pattern at scale 10: 10 unique replica names, 1 unique boot_id, 1 cluster, in all 3 repeats. This is exactly what a single-node Dedicated D8 profile must do — but it falsifies any "ACA always spreads replicas across nodes" claim. The Dedicated D8 profile co-locates by design.

- **`machine_id` is always `null` across all 1117 records.** The lab's `/diag` endpoint reads `/etc/machine-id` as a secondary kernel-context signal, but the Azure Container Apps runtime does not expose this file (the container's root filesystem does not include `/etc/machine-id`). All records carry `"machine_id": null`. The gates do NOT key off `machine_id`; the primary signal is `boot_id` (from `/proc/sys/kernel/random/boot_id`), which IS exposed because `/proc` is mounted in every container.

- **`analysis-summary.json` is a top-level JSON list, not a `{"runs": [...]}` dict.** This is a Phase A schema quirk of `analyze.py` that the Phase B `verify.sh` was rewritten to handle (Phase 11 sub-gate c and Phase 12 sub-gate e use `summary_runs = summary_data if isinstance(summary_data, list) else summary_data.get("runs", [])` to tolerate either shape). Each list entry carries the full per-run summary including `file`, `profile`, `scale_target`, `run_id`, `unique_replicas`, `unique_boot_ids`, `boot_time_clusters`, `cluster_centers_ms`, and `hit_ratio`. The `file` field is the 1:1 lookup key for reconciliation against scale JSONL filenames.

- **The `profile` field's case differs between filename and record body.** Filenames use lowercase slugs (`consumption-scale-...`, `dedicated-d8-scale-...`) while each JSONL record's `profile` field uses capitalized values (`"Consumption"`, `"Dedicated-D8"`). The Phase B `verify.sh` uses an explicit `PROFILE_FILENAME_TO_RECORD = {"consumption": "Consumption", "dedicated-d8": "Dedicated-D8"}` typed-enum map rather than a `.lower()` normalization, because the typed map fails loudly if a new profile is added without updating the mapping (a `.lower()` shortcut would silently match a misspelled profile and produce a wrong-cell verdict).

## Reproducibility

To reproduce this evidence pack against a fresh Azure subscription:

```bash
export AZ_SUBSCRIPTION="<your-subscription-id>"
export RG="rg-aca-rns-lab"
export LOCATION="koreacentral"

az group create --name "$RG" --location "$LOCATION" --subscription "$AZ_SUBSCRIPTION"

# Deploy the lab (RG + ACR + diag image + Bicep env + 2 apps on
# Consumption and Dedicated D8 workload profiles).
bash labs/replica-node-spread/deploy.sh

# Phase A — health-check the deployment (env, both workload profiles,
# both apps, /diag responds 200). Replaces the pre-Phase-B verify.sh
# which had this exact role. Sanity check only; emits no gate JSONs.
bash labs/replica-node-spread/health-check.sh
```

Expected runtime: **~50 minutes total** (~5 min Bicep deploy + ACR build, ~3 min H3 falsify gate, ~25 min Consumption scale sweep including 3 top-scale repeats with stable-replica waits, ~10 min Dedicated D8 scale sweep including 3 top-scale repeats, ~5 min analyze.py + verify.sh emission, immediate cleanup queue). Estimated cost: **<$2.50 USD** (Consumption profile with up to 30 replicas of 0.25 vCPU / 0.5 GiB for ~30 min, Dedicated D8 profile with 1 node × 8 vCPU running for ~45 min, one Basic-tier ACR, one Container Apps Environment, Korea Central). The Dedicated D8 node is the dominant cost driver because it bills per-vCPU-second on a single physical node regardless of replica count.

The Phase B `verify.sh` is a pure file processor — it reads `evidence/*.jsonl` and `evidence/analysis-summary.json` from disk and emits `evidence/10-*.json` through `evidence/13-*.json`. It does NOT call Azure, which is why the resource group can be deleted (via `cleanup.sh`) before `verify.sh` finishes the gate-emission phases. The pre-Phase-B `verify.sh` was a 5-check health-check script that ran live `/diag` probes against the deployed apps; the Phase B refactor in this commit **renamed that script to `health-check.sh`** (so Phase A operators still have the same readiness gate under a more semantically accurate name) and reassigned the `verify.sh` name to the new gate-emission processor. `trigger.sh` performs a minimal pre-flight (FQDN resolution + H3 falsify gate) but the exhaustive 5-check health-check now lives in `health-check.sh`.

## CLI versions and platform context

The captures in this pack were produced with **azure-cli 2.83.0** on Korea Central. The lab uses a single Basic-tier ACR (hash-derived suffix preserved verbatim in the evidence because it is not PII) with managed-identity ACR pull on both Container Apps. The only Azure-side resources are the ACR, the workload-profile Container Apps Environment (with both a Consumption profile and a Dedicated D8 profile), and the two Container Apps. The lab REQUIRES a region that supports **workload profiles** AND the **D8** SKU (Korea Central, East US, West Europe verified at capture time). The `/diag` workload image is `python:3.12-slim` + gunicorn baked from `labs/replica-node-spread/diag/Dockerfile` and pushed to ACR via `az acr build`. The `boot_id` signal is read from `/proc/sys/kernel/random/boot_id`, which is the kernel's per-boot random UUID — distinct kernel instances produce distinct boot_ids, while a single kernel instance (whether shared by 1 replica or N replicas via container runtime co-location) produces a single boot_id.
